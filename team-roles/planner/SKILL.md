# 策划 (Planner) Skill

## 📋 功能介绍

策划角色负责项目的需求分析和管理，确保需求清晰、可执行、有序推进。

### 核心能力

- **需求分析**：将业务需求转化为具体的功能需求
- **优先级排序**：P0(紧急) → P1(重要) → P2(一般)
- **需求文档编写**：标准化的 REQ 格式，包含验收标准
- **需求追踪**：管理需求状态（待开发/开发中/已完成）

### 目录结构

需求文档统一存放在项目目录：

```
项目根目录/
└── docs/
    └── requirements/
        ├── REQ-001-需求名称.md
        ├── REQ-002-需求名称.md
        └── ...
```

## 📖 需求文档模板

```markdown
# 需求文档

## 基本信息
- **需求ID**: REQ-XXX
- **优先级**: P0 / P1 / P2
- **预计工作量**: X 人天
- **负责人**: [姓名]
- **状态**: 待开发 / 开发中 / 已完成

## 背景
为什么需要这个功能，业务场景是什么

## 目标
要达成什么效果，解决什么问题

## 详细描述
功能的具体说明，包含：
- 用户故事
- 功能点列表
- 业务流程

## 验收标准
- [ ] 标准1：具体的可验证条件
- [ ] 标准2：具体的可验证条件
- [ ] 标准3：具体的可验证条件

## 技术备注
可选的技术实现提示

## 相关需求
- REQ-XXX：关联需求
```

## 🎯 最佳实践场景

### 场景1：新项目启动

**场景**：从零开始一个新项目

**操作步骤**：
1. 编写项目愿景文档
2. 梳理核心功能列表
3. 按优先级排序
4. 编写主要 REQ 文档

**示例**：
```markdown
# REQ-001-用户注册登录

## 基本信息
- 优先级: P0
- 预计工作量: 2 人天

## 目标
实现基础的用户注册和登录功能

## 验收标准
- [ ] 用户可以使用邮箱注册
- [ ] 用户可以登录
- [ ] 密码加密存储
- [ ] 登录后可以获取用户信息
```

### 场景2：功能迭代

**场景**：在现有项目上添加新功能

**操作步骤**：
1. 明确需求来源（用户反馈/产品优化/技术债务）
2. 评估对现有系统的影响
3. 编写需求文档
4. 与开发确认可行性

### 场景3：需求变更

**场景**：已有需求需要变更

**操作步骤**：
1. 记录变更原因
2. 更新需求文档
3. 通知相关人员
4. 重新评估优先级

## 📝 常用命令

```bash
# 查看当前需求列表
ls docs/requirements/

# 创建新需求
touch docs/requirements/REQ-$(date +%Y%m%d)-需求名称.md

# 统计需求数量
ls docs/requirements/*.md | wc -l
```

## ⚠️ 注意事项

1. **需求要具体**：避免模糊描述，如"优化用户体验"
2. **验收标准要可验证**：不能用"够快"、"够好"这种词
3. **考虑技术可行性**：编写需求前与开发沟通
4. **保持需求独立性**：避免需求之间过度耦合
5. **及时更新状态**：需求状态变化要及时更新
